## ラウンド1(2026-07-16): カメラ到着後の初期検証

### 目的
カメラ(Logitech C922 Pro Stream Webcam)を購入し実験可能になったため、
`CHECKLIST.md`/`VERIFICATION_AFTER_CAMERA.md`のうち、人間の現場作業を
必要としない項目(自動化可能な範囲)を検証する。

### やったこと
1. 環境整備: Python 3.13(`python`コマンド。`python3`はWindowsAppsの
   壊れたスタブなので使わないこと)、FFmpeg、requirements.txt一式を確認・導入
2. Mockモード回帰テスト → 問題なし
3. 実カメラでのPhase1確認(起動・クリップ生成)→ 起動はできたが…
4. 実測でフレームレートを計測 → **設定は60fpsなのに実測は約29.4fpsしか
   出ていないことを発見**
5. 原因切り分け: OpenCVのバックエンド(CAP_DSHOW vs CAP_MSMF)と
   FOURCC設定順序の組み合わせを比較実験 → `CAP_MSMF`+FOURCCを
   解像度/fpsより先に設定、で実測55〜60fpsに改善することを確認
6. `src/camera_source.py`の`RealCameraSource.start()`を修正、再検証
7. 長時間安定性テスト用の監視ハーネス(`soak_monitor.py`、health API
   ポーリング+psutilでのメモリ計測+バッファ/クリップサイズ監視)を自作し、
   30〜40分規模のテストを複数回実施
8. その過程で **Web監査サーバー(Flask開発用サーバー)が長時間稼働+
   継続アクセス時に完全にハングする** ことを発見。py-spyでスレッドダンプを
   取得し、Werkzeugのアクセスログ出力がWindows上のコンソール出力
   (colorama)の内部ロックを介して詰まっていることを特定
9. `src/web_server.py`・`central_dashboard/app.py`でWerkzeugの
   アクセスログを抑制する修正を実施、再検証(10分・30分テストで
   ハング再発なしを確認)
10. 合わせて、Webサーバーの起動失敗(ポート競合等)が発生してもこれまで
    完全にサイレントだった問題に気づいたため、`src/main.py`に起動確認
    (`_wait_for_web_server`)と警告表示・監査ログ記録を追加

### 結果(発見した問題と対応)

| 問題 | 影響 | 対応 |
|---|---|---|
| 実カメラのfps設定が実際には反映されず約29fpsしか出ない | クリップの実時間範囲が想定の約2倍になり、同期精度が崩れる。この案件の核心に関わる不具合 | `camera_source.py`: `CAP_MSMF`+FOURCC先出しに変更。修正済み・検証済み |
| Web監査サーバーが長時間稼働+アクセス継続でハング | 録画は継続するため誰も気づけないまま、iPadから監査画面に一切アクセスできなくなる | `web_server.py`/`central_dashboard/app.py`: werkzeugのアクセスログを抑制。修正済み・検証済み |
| Webサーバーの起動失敗がサイレントに握りつぶされていた | 起動時にポート競合等が起きても誰も気づけない | `main.py`: 起動後に実際にHTTP応答があるか確認し、無ければ警告+監査ログ記録。修正済み・検証済み |

### 未解決・要継続観察

- 30分テストの終盤約90秒間だけ、フレームレートが24fps程度まで一時的に
  低下する事象を1回観測。エラーは出ておらず録画・Webサーバーとも継続動作。
  USB電源管理等を疑っているが未特定。**本番前に数時間規模の連続稼働で
  再現するか確認することを推奨**(現状は30分規模の検証のみ)。

### 学んだこと・次回以降への申し送り

- このWindows環境では`python3`コマンドは使えない(WindowsAppsストアの
  スタブにリダイレクトされ動作しない)。`python`コマンドを使うこと。
- `cv2.VideoCapture.get(CAP_PROP_FPS)`は「要求した値」を返すだけで、
  実際に配信されているフレームレートを保証しない。カメラ設定を変更した
  際は、必ず実測(`read()`を一定回数呼んで所要時間を計測)で確認すること。
  ログの数値を鵜呑みにしない。
- Flaskの開発用サーバー(Werkzeug)は「開発用」と警告が出る通り、
  本番の長時間・高頻度アクセスには注意が必要。今回はアクセスログ抑制で
  回避したが、8コート運用で中央ダッシュボードが頻繁にポーリングする
  運用を考えると、将来的に本格的なWSGIサーバー(gunicorn等、ただし
  Windows対応に注意)への切り替えも検討候補。
- 長時間バックグラウンドプロセスをテストする際、`subprocess.PIPE`で
  子プロセスの標準出力を溜め込んだまま読まないと、子プロセス側の
  print()が大量に出るケースでパイプが詰まり、予期しない副作用
  (今回のハング原因の一因)を引き起こしうる。次回同様のテストをする際は
  子プロセスの標準出力を都度読み捨てるスレッドを用意するか、
  ファイルにリダイレクトすること。

### 現在の状態

- `data/clips/`・`data/buffer/`はテスト後にクリーンアップ済み
- `data/audit_log.jsonl`にはテスト中のイベントが残っている
  (本番前にクリアするか判断すること)
- 画質確認(実際の技の動きをカメラで撮影しiPadでコマ送り確認)は
  孝さん本人が実施し、「無事に動作しています」と確認済み(2026-07-16)

### 残っている人間の現場作業(未着手)

- 業者のUSBボタン到着後の同期精度実測(`VERIFICATION_AFTER_CAMERA.md`のC)
  → まず`tools/detect_keyboard_key.py`でキー名確認から
- USBケーブル抜き差しによる障害シナリオの実地確認(同D、F)
  → 現状の実装では、カメラ再接続後の自動復帰はしない設計のまま
    (`recorder.py`の`_capture_loop`はVideoCaptureを再オープンしない)
- 8コート運用時の複数台同時稼働確認(同G、複数PC・カメラが必要)

## ラウンド2(2026-07-18): クリップ永久保存機能の追加(応用性への第一歩)

### 目的

「良かったプレイを消えないように保存し、将来的に他の生徒・他の部活動とも
共有できる基盤にしたい」という孝さんの要望を受け、現行のFIFO(直近2回分
のみ保持)とは別に、監査担当が選んだクリップだけを永久に残せる仕組みを
追加する。将来的には空手専用ではなく他の部活動システムからも簡単に
繋げられる独立サービスに発展させたい、という大きな方向性が背景にある。

### 事前確認(AskUserQuestionで孝さんに確認した3点)

大きな設計判断が絡むため、実装前に以下を確認してから着手した:

1. **共有の置き場所・範囲**:「校内LAN限定の集約サーバー」を選択
   (現行の「クラウド送信は一切なし」という設計方針を維持する方向)
2. **今回のセッションでの実装範囲**:「最小機能のみ(保存ボタン+永久保存
   フォルダ)」を選択。校内LAN集約サーバーや複数部活動との連携は
   次ラウンド以降の課題として切り分けた
3. **未成年(生徒)映像の生徒間共有について学校側の方針**:「既に整理済み
   (保護者同意・校内規定を含め)。生徒間シェアも含めて設計してよい」との回答

→ この回答を踏まえ、今回は①の方向性を見据えつつ②の範囲だけを実装した。
③により、次ラウンド以降で生徒間シェアの設計に進んでよいという前提を得た
(ただし実際の同意取得方法・記録の所在などは孝さん側で管理されている前提。
 このシステム側では「誰の映像か」の識別・同意管理はまだ一切実装していない
 ことに注意)。

### やったこと

1. `src/config.py`に`SAVED_DIR`(`data/saved/`)・`SAVED_INDEX_PATH`
   (`data/saved/index.jsonl`)を追加
2. 新規モジュール`src/saved_clips.py`を作成。既存のFIFO管理
   (`clip_extractor.py`)には一切手を加えず、「`data/clips/`の確定クリップを
   `data/saved/`へコピーする」という完全に独立した処理にすることで、
   既存のクリップ生成・削除ロジックへの影響をゼロに抑えた
3. `src/web_server.py`に3つのAPIを追加:
   `POST /api/clips/<filename>/save`(保存)、
   `GET /api/saved`(一覧)、`GET /saved/<filename>`(配信)
4. `static/index.html`に「★ このクリップを保存」ボタンと「保存済み一覧」
   トグル表示を追加(選択中のクリップが既に保存済みの場合はボタンを隠す
   ことで、無意味な二重保存を防止)
5. `tests/test_save_clip.py`を新規作成し、
   「保存→一覧に出る→動画配信される→FIFOで元クリップが消えても
   保存済みコピーは残る」までを自動検証するテストを追加
6. 既存の`tests/test_pipeline_mock.py`・`tests/test_full_stack.py`が
   引き続き通ることを確認(既存ロジックへの影響がないことの裏付け)
7. Mockモードでサーバーを起動し、ブラウザ(Claude Browser)で実際に
   保存ボタン押下→保存済み一覧への反映→保存済みクリップの再生
   までをUI操作で確認(`read_page`/`computer`のref経由クリックが
   このセッションでは安定して動かなかったため、JavaScript直接実行
   経由でクリックイベントを発火させて検証した。動画の`readyState`が
   `HAVE_ENOUGH_DATA`になることまで確認済み)

### 実装中に見つけたバグ・その対応

| 問題 | 影響 | 対応 |
|---|---|---|
| 保存ファイル名に`COURT_NAME`(既定値"コート1"などの日本語)をそのまま使っていた | ファイル名がURLパスの一部になるため、日本語を含むと`urllib`等ASCII前提の処理が`UnicodeEncodeError`で落ちる(ブラウザの`fetch`/`<video src>`は自動エンコードするため気づきにくいが、他クライアントや将来のLAN集約サーバー連携で問題化しうる) | `saved_clips.py`: ファイル名にはコート名のSHA1ハッシュ先頭8文字(ASCII専用)を使うよう変更。人が読めるコート名は`index.jsonl`側の`court`フィールドに別途保持することで、UI表示上は従来通り日本語のコート名が見える |

### 学んだこと・次回以降への申し送り

- Claude Browser側の`computer`ツールで`read_page`から得た`ref`を使って
  クリックしようとすると、このセッションでは`screenshot`がタイムアウトし、
  `left_click`も"ref is stale"で失敗し続けた(定期ポーリング処理を
  止めても再現)。原因切り分けより検証を優先し、
  `javascript_tool`で直接`element.click()`を発火する方式に切り替えて
  検証を完了させた。次回同様の状況になったら、早めにこの代替手段に
  切り替えると時間を節約できる
- ファイル名やURLパスに使う識別子は、たとえ現状ASCIIでも「将来
  設定次第で日本語が入りうる」ものは早めにハッシュ化/サニタイズしておく
  べき、という教訓(`COURT_NAME`は`config.py`でユーザーが自由に
  書き換えられる値であり、実際に既定値が日本語だった)

### 現在の状態

- `data/`配下は今回のテスト・検証で作られたファイルをすべて削除済み
  (ラウンド1時点から残っていた`bar_clip_20260716_222700_853_0.mp4`
  ―孝さんが実機画質確認で使ったと思われるファイル―はそのまま残してある)
- `data/saved/`は空の状態(index.jsonlも生成前)
