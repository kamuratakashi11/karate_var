# 空手 BAR(Bunkai/Appeal Review)システム — MVP

審判の「やめ」操作(記録員によるタイマー停止)のたびに、直前10秒間の映像を
自動的に確定クリップとして保存し、監督からの抗議があった際に
監査担当者がiPad(またはPC)のブラウザ上で確認・コマ送りできるシステムです。

## 前提

- 1コート = 1PC = 1台のUSBカメラ(ロジクール製、H.264/UVC対応)
- 録画・切り出しはすべてPC上でローカル完結。クラウド送信は一切なし
- 監査画面はiPadやPCのブラウザから、同一LAN内でPCのFlaskサーバーにアクセスして閲覧
- 直近20回分(既定値、`CLIP_SLOTS`で調整可能)の「やめ」クリップを保持。
  1試合の中で複数回「やめ」が起き、試合終了後にまとめてレビューする
  運用を想定した値(空手VARの元々の用途であれば2件でも足りるが、それ以外の
  練習・他競技での利用も見据えて余裕を持たせてある)。ただし作成直後の
  クリップは、タイマーが実際に「動作中」だった累積時間が2秒(既定値)に
  達するまでは上書き削除されない(ストップボタンの連打・オフィシャルミスで
  何度押されても、保護期間中のクリップは消えず一時的に上限を超えて
  保持される。試合が再開して十分な時間動いたら通常のFIFOに戻る)
- クリップの掃除は基本的に監査画面の「ライブ一括削除」ボタンで手動で行う
  運用とし、`CLIP_SLOTS`はディスク容量が際限なく増え続けないための
  安全弁という位置づけ
- 過去の「やめ」に遡っての抗議は運用上想定していない

## 実機到着後の検証について

カメラや業者のUSBボタンが届いたら、**先にこのREADMEの手順を進める前に
`CHECKLIST.md` を参照してください**。Mock映像だけでは検証できていない
項目(特にタイマーソフトとの実際の連動が可能かどうか)をチェックリスト化しています。

カメラの基本接続(`python3 main.py --camera 0`で映像が映る)まで確認できたら、
次は `VERIFICATION_AFTER_CAMERA.md` に進んでください。長時間安定性・
同期精度の実測など、本番投入前に確認すべき項目を具体的なコマンド付きで
まとめてあります。

## セットアップ(PC側、初回のみ)

```bash
# 1. Python 3.10以降がインストールされていることを確認
python3 --version

# 2. FFmpegがインストールされていることを確認(なければ公式サイトからインストール)
ffmpeg -version

# 3. 必要なパッケージをインストール
pip install -r requirements.txt --break-system-packages
```

## 他のPC(ノートパソコンなど)への移行

このプロジェクトはGitHubで管理しています(`https://github.com/kamuratakashi11/karate_var`)。
新しいPCにも同じ内容を持っていきたい場合は、USBコピーではなくgitで取得してください。

```bash
git clone https://github.com/kamuratakashi11/karate_var.git
cd karate_var
pip install -r requirements.txt --break-system-packages
```

(Python・FFmpegが未導入の場合は、上の「セットアップ」の1・2を先に行ってください)

**注意点**

- `data/`(録画バッファ・確定クリップ・永久保存クリップ・監査ログ)は**gitの管理対象外**です
  (`.gitignore`で除外。映像データをそのままGitHub上に置きたくないための意図的な設計)。
  過去に保存したクリップ自体を新しいPCに持っていきたい場合は、`data/saved/`フォルダを
  USB等で別途手動コピーしてください。
- `src/config.py`の設定値(`COURT_NAME`、`BUTTON_KEY_NAME`、カメラのデバイス番号など)は
  **PCごとに実機で再確認が必要**です。特に`BUTTON_KEY_NAME`はUSBボタンの認識結果が
  PCやUSBポートによって変わりうるため、新しいPCでも`tools/detect_keyboard_key.py`で
  必ず確認し直してください。`COURT_NAME`は環境変数`COURT_NAME`で上書きできるため、
  複数PCで同じ`config.py`を共有しつつコートごとに変える場合はこちらを使うと
  gitの同期が楽になります。
- 今後デスクトップ側で改修した内容をノートパソコン側にも反映したい場合は、
  ノートパソコン側で `git pull` するだけで最新化できます。逆にノートパソコン側で
  変更した場合は `git add` → `git commit` → `git push` してから、デスクトップ側で
  `git pull` してください。

## 動作確認(カメラが届く前)

ダミー映像でパイプライン全体を検証できます。

```bash
cd src
python3 main.py --mock
```

起動後、ターミナルでEnterキーを押すと「やめ」操作をシミュレートし、
直近10秒のクリップが `data/clips/` に保存されます。
同じPCのブラウザで `http://127.0.0.1:5000/` を開くと監査画面が確認できます。

## カメラ到着後

USBカメラをPCに接続し、以下のように起動するだけで実カメラに切り替わります。

```bash
cd src
python3 main.py --camera 0
```

`--camera 0` の数字はOS上でのカメラのデバイス番号です。複数のカメラや
内蔵カメラがある環境では番号が変わることがあるので、映像が正しく
取得できているか(`[RealCameraSource] 実際の設定値: ...`のログ表示)を
必ず確認してください。720p/60fpsで取得できていない場合、カメラ機種の
対応状況を再確認する必要があります。

## カメラのセッティング確認(ライブプレビュー)・オートフォーカス対策

起動後、ブラウザで `http://<このPCのIPアドレス>:5000/preview` を開くと、
今カメラがどう映っているかをほぼリアルタイムで確認できます(0.4秒間隔の
更新。録画・クリップ生成には一切影響しません)。カメラの向き・画角の
調整や、下記のピント調整を行う際に使ってください。監査画面右上の
「📷 カメラプレビュー」リンクからも開けます。

試合中にオートフォーカスが迷って画面がぼやける場合は、`src/config.py`の
設定でオートフォーカスを切り、手動フォーカスに固定できます。

1. `AUTOFOCUS_ENABLED=1`(既定)のまま起動し、`/preview`を見ながら
   カメラの位置・角度を試合エリアに合わせて調整する
2. ピントが合った状態で、環境変数`MANUAL_FOCUS_VALUE`にその時の値を
   設定して再起動する(具体的な値はカメラ・距離次第。`/preview`を
   見ながら少しずつ変えて追い込む)

   ```bash
   set AUTOFOCUS_ENABLED=0
   set MANUAL_FOCUS_VALUE=(調整した値)
   python main.py --camera 0
   ```

3. `[RealCameraSource] オートフォーカス: ...`のログで、実際にオフに
   なっているか確認する(カメラ・ドライバによっては設定しても反映
   されないことがあるため、必ずログと`/preview`の見た目の両方で確認すること)

## 大会当日の運用

1. PCを起動し、USBカメラを接続してから `python3 main.py --camera 0` を実行
2. 記録員は試合中、審判が「やめ」と言うたびにEnterキー(または将来的に
   物理ボタン/フットペダル)を押す → 直近10秒が自動保存される
3. 監督から抗議があった場合、監査担当者がiPadのブラウザで
   `http://<PCのIPアドレス>:5000/` にアクセスし、該当するクリップを
   選んでスワイプでコマ送り確認する

## 業者(ネットワーク担当)への依頼事項

- PCに**固定IPアドレス**を割り当ててほしい(DHCPで毎回変わると当日困る)
- iPadとPCが同一LAN内で、**クライアント間通信(iPad→PC)がブロックされない**
  設定にしてほしい(公共Wi-FiでよくあるAP分離機能がオンだと接続できない)
- インターネット接続は不要(完全ローカル動作のため)

## 業者のタイマー用USBボタンと連動させる場合

タイマー用PCと同じPCにこのシステムを同居させる前提です。ボタンはキーボードの
**F2キー**に割り当てられていることが業者確認済みです(`src/config.py`の
`BUTTON_KEY_NAME = "f2"`に設定済み)。OSレベルのグローバルキーボード監視
(pynput)で、タイマーソフトに触れずにF2キー入力だけを横取りします。

1. (念のための最終確認)ボタンをPCに接続した状態で診断ツールを実行し、
   実際にF2として表示されるか確認しておくと安心です

   ```bash
   python3 tools/detect_keyboard_key.py
   ```

   ボタンを押して `キー名 = 'f2'` と表示されればOK。もし違う表記(例: 大文字の
   `F2`など)で表示された場合は、`src/config.py`の`BUTTON_KEY_NAME`をその
   表記に合わせて修正してください。

2. 通常起動時に `--input-mode button` を付けて起動する

   ```bash
   python3 main.py --camera 0 --input-mode button
   ```

   これで、業者のボタン(F2キー)を押すと(タイマー側の停止動作とは独立して)
   このシステム側でも直近10秒のクリップ確定が同時に走るようになる。

### タイマーがトグル式(F2で開始/停止を兼用)であることへの対応

F2は1つのボタンでスタート/ストップを交互に切り替えるトグル式であることが
確認されている。そのためシステム内部でF2の押下回数を数えて追跡し、
「ストップ」に切り替わった瞬間だけクリップを確定させる。

ただしこの追跡方式は、押下の取りこぼしなどで一度でもズレると、以降
スタート/ストップの判定が反転したまま誤動作し続けるという構造的な
弱点がある。そのため、**緊急再同期ボタン**を別のキーに割り当てる。

- `src/config.py` の `BUTTON_EMERGENCY_KEY_NAME` に設定する(`tools/detect_keyboard_key.py`
  で確認したキー名を使う。業者のF2ボタンとは別の、こちらのシステム専用のキー/ボタンを
  用意すること)
- このボタンが押されると、追跡状態に関係なく**必ず**直近10秒のクリップを確定し、
  同時に内部状態を「今は確実に停止中」に強制リセットする
- 記録員・監査は、タイマーの実際の表示とシステムの追跡状態(監査画面に
  常時表示される「● タイマー追跡状態: 動作中/停止中」)を見比べて、
  ズレを感じたらこの緊急ボタンを押すことで復帰できる

`src/config.py` の `TIMER_INITIAL_STATE` は、システム起動時点でタイマーが
停止中であることを前提にしている(既定値`"stopped"`)。起動前に、実際の
タイマーが本当に停止中であることを確認してから起動すること。

### タイマーとの同期精度について

録画の内部断片形式にMPEG-TS(.ts)を採用しており、F2キーが押された
瞬間に「今まさに書き込み中の断片」も含めて切り出せるようにしてある。
これにより、タイマーが止まった瞬間と切り出される映像の終端のズレは
最大でも1フレーム程度(60fpsならおよそ16.7ms)に収まる設計になっている。
(mp4形式のまま書き込み中の断片を除外していた場合、最大でSEGMENT_SECONDS
(既定2秒)分のズレが出てしまうため、この方式は採用していない)

**破損対策(ラウンド6、孝さんの発案による設計)**: 「書き込み中の断片も
安全に読める」という前提は、実機検証の結果、必ずしも正しくないことが
分かった(録画側が書き込んでいる最中のファイルを同時に読みに行くと、
書きかけの不完全なデータを掴んでしまい、映像が破損することがある。
実機では従来方式でほぼ毎回発生していた)。

そこで、「やめ」の瞬間に書き込み中だったセグメントが完全に書き終わる
(=次のセグメントに切り替わる)まで待ってから読みに行く方式にした。
待ち時間は最大でも`SEGMENT_SECONDS`秒程度で済み(実測では1秒前後)、
**同期精度を落とさずに**読み取り競合そのものを避けられる。実機で
5回連続テストし、全て1回目から正常(破損なし)を確認済み。

万一それでも壊れていた場合の保険として、生成直後にデコード検証も行い、
なお壊れていたら「書き込み中の断片を除外した安全な方式」で自動的に
作り直す(この場合のみ同期精度が最大SEGMENT_SECONDS秒ほど犠牲になる。
作り直しが発生した場合は監査ログに`clip_corruption_fallback`として記録される)。

※この待ちがあるため、`extract_on_yame()`は呼び出し元を最大数秒ブロックする。
物理ボタン(`--input-mode button`)使用時は、この待ちがpynputのキーボード
フックのコールバックスレッドを直接ブロックしないよう、内部で別スレッドに
逃がして実行している(直接ブロックすると、Windowsがフックを応答なしと
みなして無効化してしまう危険があるため)。

※タイマーソフト自体には一切手を加えていない。あくまでOSレベルで
「同じボタンの入力信号」を横取りして拾っているだけの方式。

## クリップの永久保存(良かったプレイを残す)

ライブの「やめ」クリップは通常FIFOで自動削除されますが、監査画面で
「★ このクリップを保存」ボタンを押すと、そのクリップは `data/saved/` に
コピーされ、以後FIFOの対象から外れて消えずに残り続けます。

- 監査画面の「保存済み一覧を表示」から、これまで保存したクリップを
  いつでも見返せます(通常のクリップと同じスワイプ・コマ送り操作が可能)。
- 保存は元クリップの**コピー**であり、`data/clips/` 側のFIFO動作には
  一切影響しません。
- 保存されたファイル名にはコートを識別する短い符号(COURT_NAMEのハッシュ値)
  が入りますが、これは日本語のコート名をそのままURLの一部に使うと
  文字コード関連の不具合が起きやすいためです。人が読める本来のコート名は
  `data/saved/index.jsonl` のメタ情報(`court`欄)の方に記録されます。
- 現時点ではPC1台(コート1台)内のローカル保存のみです。他のコートや
  他の生徒との共有、他の部活動システムとの連携は次段階の検討課題です
  (下記「将来検討」を参照)。

## 試合ごとの管理(「次の試合へ」ボタン)

空手VARに限らず、1試合の中で複数回「やめ」が起き、試合終了後にまとめて
レビューしたい運用や、複数試合をまたいで後から振り返りたい運用に対応する
ための機能です。

- 監査画面上部に現在の試合番号(「試合1」など)と「次の試合へ →」ボタンが
  表示されます。記録員が試合の区切りでこのボタンを押すと、番号が1つ進み、
  それ以降に作られるクリップには新しい試合番号が付きます
- PC側のキーボードで**Nキー**を押しても同じ動作をします(`--input-mode enter`
  でも`button`でも常に有効)。`--input-mode enter`使用時にNキーを押すと
  「n」の文字が入力行に残ることがありますが、続けてEnterを押しても
  「やめ」は誤発火しません(Nキー自体はEnter待ちとは別経路で即座に
  処理されるため)
- ライブ一覧・保存済み一覧のどちらも、試合番号ごとにグループ分けして
  表示されます(新しい試合が上)
- 試合番号は現状「記録員が手動で次へ進める」自動採番方式です。将来的に
  空手タイマーシステム側が保持している試合番号をそのまま読み取って
  同期する方式に差し替える予定があるため、試合番号の設定経路は
  `ClipExtractor.set_match()` の1箇所にまとめてあります
- 「ライブ一括削除」ボタンを押すと、その時点でライブに残っている
  未保存のクリップをまとめて削除できます(確認ダイアログあり)。
  **`data/saved/` に保存済みのクリップには一切影響しません。** 試合の
  レビュー・保存が一通り終わったタイミングで、次の試合に備えて
  一覧を整理するために使う想定です

## 8コート運用時の中央死活監視ダッシュボード

各コートのPC(1コート1台)は今まで通りの手順で起動する。加えて、運営本部の
PC1台で以下を実行すると、8コート分の「録画が正常に動いているか」だけを
一覧表示できる(映像データは一切やり取りしない、軽量な死活監視専用の画面)。

1. 業者に各コートPCへ固定IPを割り当ててもらう
2. `central_dashboard/courts_config.json` に、各コートの名前とIPアドレスを記入する
3. 運営本部PCで以下を実行する

   ```bash
   cd central_dashboard
   python3 app.py
   ```

4. ブラウザで `http://<運営本部PCのIP>:6000/` を開くと、緑(録画中)・赤(録画停止)・
   グレー(接続不可)で8コート分の状態が一覧表示される

- 複数カメラの冗長化(1コートに複数台のカメラを持たせる場合)
- 保持枠数(既定20、環境変数CLIP_SLOTSで上書き可)・保護時間(現在2秒)の
  運用中の可変設定(値の変更自体は再起動不要な仕組みにはなっていない)
- ディスク空き容量が少なくなった際の警告(現状は監視の仕組みなし。
  試合ごとの「ライブ一括削除」ボタンで定期的に掃除する運用が前提)
- 試合番号を空手タイマーシステム側から自動で読み取る連携
  (現状は記録員が「次の試合へ」ボタンで手動送りする方式。差し替えの
  受け口として`ClipExtractor.set_match()`は用意済み)
- 中央ダッシュボードからの各コートクリップ映像の直接閲覧(現状は死活監視のみ。
  クリップ本体の確認は引き続き各コートの監査担当がその場で行う設計)
- **複数コート・複数生徒間での保存クリップの共有**(校内LAN上の集約サーバーに
  各コートの`data/saved/`を送る、というイメージを検討中。中央死活監視
  ダッシュボード(`central_dashboard/`)と同じ「別プロセス・別PCで動く
  疎結合サービス」のパターンを踏襲する想定)
- **空手以外の部活動での再利用**(このシステムの「ボタン押下→直近数秒を
  確定保存」という仕組み自体は空手固有ではないため、将来的に他の運動系
  部活動のタイマー/システムからも同じ仕組みに簡単に接続できるようにする)
- 保存クリップの閲覧権限・生徒ごとの識別・タグ付け・検索(現状は
  「保存済み一覧」を時系列で見るだけの最小機能)
